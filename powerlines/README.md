# Информационная система электросетевой компании

Реализованная система упрощена относительно реальных ГИС электросетевых компаний. Она показывается только принципиальное строение подобных систем, но может использоваться для MVP.

## Состав репозитория

| Путь | Содержание |
|------|------------|
| `sql/01_schema_tables.sql` | Домены, таблицы, комментарии к таблицам и столбцам |
| `sql/02_indexes.sql` | Индексы (в т.ч. частичный и GIN по JSONB) |
| `sql/03_functions.sql` | Пользовательские функции |
| `sql/04_procedures.sql` | Хранимые процедуры |
| `sql/05_views.sql` | Представления для отчётов |
| `sql/06_triggers.sql` | Триггеры проверки и аудита |
| `sql/07_security.sql` | Роли `powerlines_readonly` и `powerlines_app`, привилегии |
| `sql/08_queries_reporting.sql` | Аналитика (CTE, подзапрос) и запросы мониторинга |
| `data.sql` | Демонстрационные данные |
| `database_structure.md` | Текстовое описание таблиц и связей |

Во всех файлах были добавлены комментарии для понимания того, что делает тот или иной скрипт.

В скриптах создающих таблицы добавлены комментарии под каждый столбец, чтобы понимать их назначения.

## Порядок применения в pgAdmin / DBeaver / psql

**Обязательно нужна PostgreSQL с PostGIS, т.к. в таблицах используются координаты.**

1. Создать пользователя (например, otus, лучше с правами супер-пользователя, чтобы не было проблем с установкой плагинов и мониторингом производительности);
2. Создать пустую базу (например, `powerlines`, кодировка UTF-8);
3. Дать созданному ранее пользователю права на созданную базу;
4. Последовательно выполнить скрипты `01` - `07`. Это создаст все нужные таблицы, индексы, функции, процедуры и т.п.;
4. Выполнить скрипт `data.sql`, который наполнит таблицы данными (чистая генерация ИИ, поэтому к реальности они отношения не имеют);
5. Если нужны примеры работы, то можно выполнить запросы, которые есть в `08_queries_reporting.sql`.

## Оптимизация

Для более оптимальной работы с БД используется несколько типов индексов:
- **B-tree** по внешним ключам и типовым фильтрам (например, `region_id`, `line_id` и т.п.), ускорение стандартных выборок по ключам;
- **GiST** по геометрии PostGIS (`supports.base_geom`, `conductors.end_a_geom`), даёт прирост для запросов, связанных с пространственными расчётами;
- **Частичный индекс** `idx_outages_open` используется только по незакрытым отключениям. Меньше строк в индексе при частых выборках «активных» событий;
- **GIN** по `JSONB` (например, `extra_spec` у ВЛ, `meta` у документов) — ускорение поиска по ключам JSON при аналитике;

В файле `08_queries_reporting.sql` приведён пример использования `EXPLAIN ANALYZE` для проверки плана (можно запустить и проверить вручную).
При необходимости можно удалить какие-то индексы и сделать замеры без них. Хотя на небольшой выборке особого прироста в производительности можно и не заметить.

## Мониторинг

В `08_queries_reporting.sql` используются `pg_stat_user_tables` и `pg_stat_activity` для оценки активности и объёма данных. При необходимости в кластере можно включить расширение `pg_stat_statements`.

## Безопасность

Создаются роли:

- `powerlines_readonly` — только `SELECT` (и вызов функций для отчётов).
- `powerlines_app` — полный CRUD по данным и вызов процедур.

## Дополнительно

- **JSON**: поля `power_lines.extra_spec` и `documents.meta` для гибких атрибутов без изменения схемы.
- **PostGIS**: координаты — `geometry(Point, 4326)`; расстояния в функциях и процедурах считаются через приведение к `geography` (эллипсоид WGS84).
